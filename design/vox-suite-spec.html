00

Intent

Why this exists and how it should feel.

Today the two tools look unrelated: VoxAnalysis is already a dark, cyan/mono "studio" surface with a live signal‑chain and meters; VoxPolish is a plainer charcoal editor on a lone yellow accent. They now share one repo — this system makes them share one face.

The governing metaphor is the reference image: a mastering control room. Not skeuomorphic knobs — the behaviour of instruments. Three principles carry the whole system:

  • Always instrumented. The interface continuously reports state — idle, uploading, isolating, analyzing, rendering, done — through lights, meters and a running signal chain. Nothing happens silently.
  • Cyan is the signal. One confident accent for everything live and interactive. Colour means something; it is never decoration.
  • Calm until it matters. Ambient motion stays low; amber and red are held in reserve so that when something needs you, it genuinely stands out.
01

Structure & navigation

One shell. Two modes. An optional fused path.

A single persistent shell wraps both tools: a command bar (brand, system state, elapsed, master level), a mode switch, the signal chain, and the shared telemetry rail. Only the working area between the rails swaps per mode.

Polish

Repair, level and subtly tune a take. Owns the module rail + waveform editor.

  • Signal modules: Separation, Dynamics, Gate, Breath, Sibilance, Tune
  • Waveform scope · region editing · re‑render

Analyze

Measure, score and compare against the verified original. Owns the readout rail + pitch scope.

  • Live readout: pitch, cents, range, confidence, H1–H8
  • Calibrated scorecard · trouble spots · coaching

Fused workflow · opt‑in

Upload a take once and run the whole pipeline end‑to‑end; the same telemetry follows it across both engines. Offered as a choice, never forced — Polish and Analyze remain usable on their own.

Upload→Isolate→Analyze / score→Polish / repair→Export + report
Shell anatomy Command bar → Mode switch → Signal chain → [ left rail · scope · telemetry rail ] → Processing bar. The three horizontal bands are constant across modes; the rails' contents change.
02

Color system

Cyan-forward, on a blue-biased black.

Neutrals are a near-black with a slight blue bias — chosen, not a default grey — so cyan reads as emission against it. Cyan is the sole brand accent. Amber, green and red are semantic only: they mean warning, healthy/complete, and critical respectively, and must not be used to "brand" or decorate.

DO
  • Use cyan for every active/interactive/live element.
  • Let green mean "complete/healthy" consistently across both modes.
  • Keep violet narrow — Analyze‑mode identity and the singer pitch trace.
DON'T
  • Reintroduce VoxPolish yellow anywhere.
  • Use amber as an accent or edge‑light — it now signals caution.
  • Mix cyan and violet as co‑equal accents on one surface.
One open call The reference leans hard on amber edge‑lighting. Under "cyan primary, amber = warnings," I'm proposing the ambient glow be cyan, with amber strictly for caution states. If you want a warm structural glow purely as decoration, we'd carve out a fourth neutral‑warm token so it doesn't collide with the warning meaning. Flagging rather than deciding.
03

Typography

Two voices: display and instrument.

No web font is loaded (the Artifact/app CSP blocks font CDNs, and a silent fallback is worse than an honest system stack). The pairing is deliberate rather than a compromise: a heavy grotesque for headline weight, a monospace for every readout — because instrument panels are monospaced.

RoleFamilyTreatmentUsed for
Displaysystem-ui, 800tight -0.03em, balanceHeadlines, big scores, the pitch note
Bodysystem-ui, 400–600~65ch measure, 1.6 lineProse, help, descriptions
Instrumentui-monospaceuppercase, .06–.18em tracking, tabular-numsLabels, timecode, dB, Hz, state, logs, chain

Rule of thumb: if it's a measurement or a machine word, it's mono and usually uppercase. If it's language for a human, it's the sans.

04

Telemetry & motion

The living instrument — your core ask.

"Flashing lights and knowing when things are happening" becomes a small, strict vocabulary of live components. Each maps to real job stages both apps already emit (VoxAnalysis has a stage stream today; VoxPolish exposes pipeline steps) — the lights are driven, never faked.

▹State LED — the system's pulse

One canonical status light, mirrored wherever state matters (command bar, chain, rec dot). Colour + animation encode the state together:

State LED · vocabulary
STANDBY· cyan, slow pulse WORKING· amber, blink COMPLETE· green, settle ALERT· red, fast blink ANALYZE· violet, mode tint

Note the one nuance vs. the reference: active work reads amber here because amber = "attention/in‑progress," while done is green. That keeps a glance meaningful.

▹Motion rules

  • Ambient is subtle: waveform drift, meter float, LED pulse. Nothing races; the room breathes.
  • Events are crisp: a stage completing snaps green; an alert blinks fast. Change draws the eye, steady state does not.
  • Bound to truth: a meter moves because a level moved; a step lights because that stage started.
  • Respect prefers-reduced-motion: all loops stop; components render a representative static state (this page already does).

06

Screens & states

What we'll design, screen by screen.

  • Shell / idle: deck armed, STANDBY pulse, empty scope reading "awaiting take," chain dim.
  • Intake / upload: bracketed drop target, performer/song fields (Analyze), Polish/Tune choice, upload progress feeding the chain.
  • Polish · working & editor: module rail live, waveform with editable regions, meters + gauges moving, chain advancing — ending in a render → export tray (polished vocal, remix, isolated stem, editable edit_document.json; multi‑select + zip). The export is the payoff, so the tray lights up cyan and auto‑reveals on completion.
  • Analyze · working & report: pitch scope, live readout rail, then the calibrated scorecard, trouble‑spot timeline, comparison, coaching racks — with its own Export report tray (report PDF + raw measurements JSON; no rendered audio).
  • Fused: a single continuous run — one chain spanning isolate → analyze → polish → export.
  • Alert / error: red LED, amber banner with plain‑language cause + fix, meters flagged if clipping.
  • Reduced‑motion & mobile: static representative states; rails stack, transport docks to the bottom (VoxAnalysis already does this — we keep it).
07

Build approach

How the spec becomes code.

  • Shared design layer: one vox-tokens.css (the palette above) + vox-kit.css (shell, LED, chain, meters, buttons) imported by both front‑ends. Single source of truth.
  • Canvas for instruments: scope waveform, pitch lane, meters and gauges are canvas‑drawn (crisp, cheap, animatable) — VoxAnalysis already renders its scope this way.
  • Telemetry binding: a tiny shared client that subscribes to each engine's stage/progress events and drives the state LED, chain, processing bar and log. Define the event contract once; both back‑ends emit to it.
  • Retire yellow: replace VoxPolish's --accent:#e8d44d and its charcoal tokens with the shared set; re‑map module colours onto the new palette.
  • Accessibility: visible focus rings (cyan, offset), 44px targets, aria-pressed/role="status" live regions, motion opt‑out — all carried from the current analyzer.
  • Sequence: tokens + kit → shell + telemetry client → Polish reskin → Analyze reslot → fused path → polish pass.
